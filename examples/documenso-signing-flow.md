# Recipient Signing

**Trigger:** Recipient clicks the signing link in their invitation email (or opens a direct link)
**Actors:** Recipient (Signer, Approver, Viewer, or Assistant)
**Preconditions:** Document is in "Pending" status and it is the recipient's turn to act (if sequential signing is enabled)

## Happy Path

1. Recipient opens the signing link
   - Screen: `/sign/{token}`
   - Inputs: None (token in URL identifies the recipient)
   - System response: The document PDF is displayed with the recipient's assigned fields highlighted. An audit log entry records that the document was viewed.

2. Recipient completes their assigned fields
   - Screen: `/sign/{token}` — PDF viewer with interactive fields
   - Inputs: Varies by field type:
     - Signature: Draw, type, or upload a signature image
     - Free Signature: Freehand drawing on the field area
     - Initials: Draw or type initials
     - Name: Text entry (may auto-fill from account)
     - Email: Text entry (may auto-fill)
     - Date: Date picker
     - Text/Number: Free text or numeric entry
     - Radio/Checkbox: Selection
     - Dropdown: Choose from predefined options
   - System response: Fields are filled and visually marked as completed

3. Recipient submits their signature
   - Screen: `/sign/{token}`
   - Inputs: Recipient confirms submission
   - System response: Recipient's signing status changes to "Signed". User is redirected to `/sign/{token}/complete` showing a completion celebration screen with a checkmark icon.

4. (If all recipients have completed) Document is sealed
   - System response: Background job `internal.seal-document` runs. The PDF is cryptographically signed. Document status changes to "Completed". All parties receive a "Document Completed" email with a download link.

## Branching Paths

### Authentication Required
- Condition: Document has access authentication enabled (Account, Passkey, 2FA, or Password)
- Steps: Before seeing the document, recipient must pass the required authentication check. For Account auth, they must be signed in. For 2FA, they receive a one-time code via email. For Passkey, they use their hardware key. For Password, they enter the document password.
- Outcome: Once authenticated, the signing flow proceeds normally.

### Recipient is Not Next in Signing Order
- Condition: Sequential signing is enabled and a prior recipient hasn't completed yet
- Steps: Recipient opens the link
- Outcome: Recipient is redirected to `/sign/{token}/waiting` showing a waiting page with a clock icon.

### Recipient is an Approver
- Condition: Recipient's role is Approver
- Steps: Approver reviews the document and approves it
- Outcome: Approval is recorded. Flow continues to next recipient or document completion.

### Recipient is a Viewer
- Condition: Recipient's role is Viewer
- Steps: Viewer opens the document and marks it as viewed
- Outcome: View is recorded. Flow continues.

### Recipient is an Assistant
- Condition: Recipient's role is Assistant
- Steps: Assistant fills in fields on behalf of the primary signer. The signer still needs to sign separately.
- Outcome: Fields are pre-filled for the signer.

### Mid-Session Expiration (Opened But Not Yet Signed)
- Condition: The signing link expires WHILE the signer has the page open and is filling in fields, but has not yet submitted
- Steps:
  1. Signer opened the page before expiration — passed the initial check, sees the document normally
  2. Expiration time passes silently — there is NO client-side timer, polling, or real-time warning
  3. Signer continues filling fields unaware that the deadline has passed
  4. When the signer tries to submit a field or complete signing, the system re-checks expiration at submission time
- Outcome: The submission fails. For individual field submissions, the signer sees a red toast notification: "An error occurred while signing the document." [RENDERED — document-signing-text-field.tsx error handler]. For the final completion dialog, the error is not specifically caught (only 2FA errors are handled), so the submission fails silently or falls through to a general error boundary. The signer is NOT redirected to the `/sign/{token}/expired` page — they remain on the signing page with their work lost.
- Product risk: **HIGH** — The signer receives no advance warning, sees only a generic error message (not mentioning expiration), and may not understand why signing failed. Their partially completed fields are not saved.

### Recipient Rejects the Document
- Condition: Recipient clicks the reject option
- Steps: Recipient provides a rejection reason
- Outcome: Recipient is redirected to `/sign/{token}/rejected` showing a red X icon. Document status changes to "Rejected". Rejection emails are sent to the sender and other recipients.

## Error States

| Error | Trigger | User Sees | Recovery |
|-------|---------|-----------|----------|
| Signing window expired (before page load) | Recipient opens link after expiration date | Redirected to `/sign/{token}/expired` showing a timer-off icon and the document title [RENDERED — expired.tsx] | Contact the document sender to resend |
| Signing window expired (mid-session) | Expiration passes while signer is on the page, then they try to submit a field | Toast: "An error occurred while signing the document." [RENDERED — document-signing-text-field.tsx]. Backend throws RECIPIENT_EXPIRED with message "Recipient signing window has expired" [BACKEND-DEFINED — not surfaced to user]. Signer is NOT redirected to the expired page. | Signer must reload the page (which then redirects to expired page) or contact sender |
| Signing window expired (completion attempt) | Expiration passes, signer tries to finalize via the complete dialog | Error is not specifically caught in the completion dialog (only 2FA errors are handled) [RENDERED — document-signing-complete-dialog.tsx]. Submission fails without a clear message. | Signer must reload the page or contact sender |
| Document not found | Invalid or tampered token | HTTP 404 page [RENDERED] | Request a new link from the sender |
| Document is still a draft | Token belongs to an unsent document | HTTP 404 page [RENDERED — loader checks DocumentStatus.DRAFT] | Wait for sender to send the document |
| Authentication failed | Recipient fails 2FA or password check | TWO_FACTOR_AUTH_FAILED error [BACKEND-DEFINED — frontend rendering depends on auth page implementation] | Retry authentication or request help |
| Already signed | Recipient already completed their part | Redirected to `/sign/{token}/complete` [RENDERED] | No action needed |

## Side Effects

- Audit log entry when document is opened/viewed
- Audit log entry when each field is signed
- Email notification to document sender when recipient signs (`send-recipient-signed-email`)
- Webhook event DOCUMENT_SIGNED fired when a recipient completes
- Webhook event DOCUMENT_RECIPIENT_COMPLETED fired for each recipient completion
- If all done: `internal.seal-document` job seals the PDF; DOCUMENT_COMPLETED webhook fires; completion emails sent to all parties

## Access Control

| Action | Allowed Roles | Denied Behavior |
|--------|---------------|-----------------|
| View signing page | Recipient with valid token | 404 page if token is invalid |
| Submit signature | Recipient whose turn it is | Redirected to waiting page if not their turn |
| Access authenticated document | Recipient who passes auth check | Authentication challenge page shown |
